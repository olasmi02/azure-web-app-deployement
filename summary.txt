Azure Node.js App — Deployment Summary
=======================================

Runtime Stack & Operating System
--------------------------------
- Runtime: Node.js 22 LTS (with Express 4.x)
- Operating System: Linux
- Justification: Node.js 22 LTS was selected to leverage the latest long-term support release, ensuring security patches, performance optimizations (such as the V8 engine upgrades), and modern JavaScript features. Linux was chosen as the hosting OS because it runs Node.js applications natively inside lightweight Docker containers. This offers faster deployments, lower resource overhead, better NPM package compatibility, and is more cost-effective than Windows hosting.

App Service Plan Pricing Tier & Region
--------------------------------------
- Pricing Tier: Basic B1 (1 Core, 1.75 GB RAM)
- Region: West Europe
- Justification (Pricing Tier): The B1 plan provides dedicated compute resources (unlike shared plans like F1/Free which throttle CPU after 60 minutes/day). It features "Always On" to prevent cold starts, custom domain and SSL binding support, and basic scaling. This ensures the app is highly available and responsive.
- Justification (Region): West Europe was chosen to minimize network latency for European developers and users, comply with EU data residency regulations (such as GDPR), and ensure access to Azure's robust regional infrastructure and feature availability.

Deployment Method
-----------------
GitHub Continuous Deployment (CI/CD) integration via the Azure Portal Deployment Center.
- Workflow: The App Service was linked directly to the GitHub repository's main branch.
- Automatic Build: Upon every commit pushed to 'main', a GitHub Actions workflow (automatically generated by Azure) compiles and builds the app using Azure Oryx. Oryx detects the package.json, runs `npm install`, and packages the app.
- Deployment: The build artifact is zipped and deployed to the App Service with zero-downtime deployment slots (if configured) or direct deployment, keeping the app continuously running.

Startup Command
---------------
Azure automatically detects the "start" script in package.json and executes it:
  "scripts": { "start": "node app.js" }
No custom startup command is needed.

Custom 404 Error Page
---------------------
Express middleware is registered at the end of the routing chain to intercept non-existent routes, return an HTTP 404 status code, and render a premium-designed, responsive 404.ejs template indicating the path that could not be resolved.

Environment Variables Configured
---------------------------------
- APP_ENV = production
- APP_NAME = Azure Node App
Both are securely stored as Application Settings under Configuration in the Azure Portal, exposing them to the runtime container as standard OS environment variables (process.env.APP_ENV and process.env.APP_NAME) without hardcoding them in source code.

Monitoring and Diagnostics Guide
--------------------------------
1. Metrics Blade: Accessible via Azure Portal -> App Service -> Metrics. Enables monitoring of real-time CPU percentage, memory working set, HTTP response times, and total requests.
2. Log Stream: Accessible via Azure Portal -> App Service -> Log Stream. Shows live console output (stdout/stderr) from Node.js (e.g. app.listen port messages or error stack traces).
3. Application Insights: Provides deep performance telemetry, end-to-end transaction tracing, and dependency mapping. Accessible from the Application Insights blade or linked resource group.

Scaling Capabilities
--------------------
- Manual Scale-Out: B1 allows manual scaling up to 3 VM instances to distribute incoming HTTP traffic.
- Scale-Up (Vertical): Easily scale to B2, B3, or Premium (P1v3/P2v3) plans to get more CPU cores and RAM.
- Auto-Scale: The B1 tier does NOT support automated rules-based scaling (e.g. scale out when CPU > 70%). For auto-scaling, the plan must be scaled up to Standard (S1) or higher.
- When to Scale Out: When concurrent user traffic increases, resulting in high CPU/memory utilization or elevated page response times.
- When to Scale Up: When the app requires more memory/CPU than 1.75 GB/1 core, or requires advanced features like deployment slots, virtual network integration, or automated backup capabilities.
